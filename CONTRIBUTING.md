## 1. Кодекс поведения

Неприемлемое поведение включает:

- Оскорбления по признаку пола, гендерной идентичности, сексуальной ориентации, инвалидности, внешности, размера тела, расы, национальности или религии

- Публикация чужой конфиденциальной информации
- Преследование или нежелательное внимание
- Любые другие формы харассмента

### Нарушения

Если вы стали свидетелем или жертвой неприемлемого поведения, сообщите об этом по адресу contact@example.com. Все жалобы будут рассмотрены и рассследованы.

## 3. Процесс внесения вклада

### Шаг 1: Установка окружения для разработки

1. Сделайте fork репозитория на GitHub
   ```bash
   Нажмите кнопку "Fork" на странице репозитория
   ```

2. Клонируйте ваш fork
   ```bash
   git clone https://github.com/yourusername/time-series-analysis-course.git
   cd time-series-analysis-course
   ```

3. Добавьте upstream репозиторий
   ```bash
   git remote add upstream https://github.com/original-author/time-series-analysis-course.git
   ```

4. Создайте виртуальное окружение
   ```bash
   python3 -m venv venv
   source venv/bin/activate  # на Windows: venv\Scripts\activate
   ```

5. Установите зависимости
   ```bash
   pip install -r requirements.txt
   pip install pytest black flake8 jupyter
   ```

### Шаг 2: Создание ветки для работы

Перед тем как начать работу, обновитесь из upstream:

```bash
git fetch upstream
git checkout -b fix-issue-123 upstream/main
```

Название ветки должно быть описательным:

- Для исправления ошибок: `fix/fix-description`
- Для новых функций: `feature/feature-name`
- Для улучшений документации: `docs/improvement-description`
- Для улучшений кода: `refactor/refactor-description`

Примеры:
- `fix/typo-in-module-3`
- `feature/new-visualization-example`
- `docs/clarify-stationarity-concept`
- `refactor/improve-arima-code`

### Шаг 3: Внесение изменений

При внесении изменений следуйте этим правилам:

#### Для Python кода

1. Соблюдайте PEP 8 стиль кода
   ```bash
   black your_file.py
   flake8 your_file.py
   ```

2. Добавляйте комментарии и docstrings
   ```python
   def calculate_moving_average(series, window):
       """
       Вычисляет простое скользящее среднее.
       
       Parameters:
           series: pandas.Series - временной ряд
           window: int - размер окна
       
       Returns:
           pandas.Series - результат сглаживания
       """
       return series.rolling(window).mean()
   ```

3. Используйте понятные имена переменных
   ```python
   # Хорошо
   time_series_data = pd.read_csv('data.csv')
   
   # Плохо
   ts = pd.read_csv('data.csv')
   ```

#### Для Jupyter Notebooks

1. Очищайте output перед commit
   - Kernel > Restart & Clear Output

2. Используйте markdown для объяснений
   - Структурируйте с помощью heading'ов
   - Используйте LaTeX для формул

3. Добавляйте комментарии в код
   ```python
   # Инициализируем параметры модели
   alpha = 0.1
   beta = 0.05
   ```

4. Проверяйте, что код работает
   - Запустите все ячейки
   - Убедитесь в отсутствии ошибок

#### Для документации

1. Используйте markdown стиль
2. Придерживайтесь консистентного форматирования
3. Проверяйте ссылки
4. Используйте спеллчекер

### Шаг 4: Тестирование

Перед отправкой Pull Request убедитесь:

1. Код работает без ошибок
2. Примеры выполняются корректно
3. Нет конфликтов с существующим кодом
4. Документация актуальна

Для запуска тестов:
```bash
pytest tests/
```

### Шаг 5: Коммиты

Пишите ясные и описательные сообщения коммитов:

```bash
git commit -m "Fix: исправлен расчет ADF-статистики в модуле 3"
```

Формат коммита:
```
<Тип>: <Описание>

<Опциональное подробное объяснение>
```

Типы коммитов:
- `fix`: исправление ошибок
- `feature`: добавление новых функций
- `docs`: изменения в документации
- `style`: форматирование кода (без функциональных изменений)
- `refactor`: переструктурирование кода
- `test`: добавление или обновление тестов
- `chore`: обновление зависимостей, конфиг

Примеры:
- `git commit -m "fix: исправлена опечатка в объяснении стационарности"`
- `git commit -m "feature: добавлен пример с Box-Cox преобразованием"`
- `git commit -m "docs: улучшено объяснение метода Хольта"`

### Шаг 6: Отправка изменений

1. Обновитесь из upstream перед push
   ```bash
   git fetch upstream
   git rebase upstream/main
   ```

2. Отправьте на ваш fork
   ```bash
   git push origin fix-issue-123
   ```

### Шаг 7: Создание Pull Request

1. Откройте GitHub и перейдите в ваш fork
2. Нажмите "New Pull Request"
3. Убедитесь, что:
   - Base branch: `main` исходного репозитория
   - Compare branch: ваша ветка

4. Заполните описание PR согласно шаблону:

```markdown
## Описание

Краткое описание изменений.

## Тип изменения

- [ ] Исправление ошибки
- [ ] Новая функция
- [ ] Улучшение документации
- [ ] Другое

## Изменения

- Что было изменено
- Почему было изменено
- Как это влияет на пользователей

## Связанные Issue

Closes #123

## Тестирование

Как вы тестировали эти изменения?

## Чек-лист

- [ ] Код следует PEP 8
- [ ] Комментарии добавлены
- [ ] Документация обновлена
- [ ] Нет конфликтов с main веткой
- [ ] Тесты прошли успешно
```

5. Нажмите "Create Pull Request"

### Шаг 8: Рассмотрение и итерация

1. Поддерживайте диалог с рецензентами
2. Вносите запрошенные изменения
3. Заново отправьте изменения на ту же ветку
4. Pull Request обновится автоматически

### Шаг 9: Слияние

После одобрения рецензента ваш Pull Request будет объединен с main веткой.

## 6. Решение конфликтов

Если у вас возникли конфликты при merge:

1. Получите обновления из upstream
   ```bash
   git fetch upstream
   git merge upstream/main
   ```

2. Разрешите конфликты в редакторе
3. Добавьте разрешенные файлы
   ```bash
   git add .
   ```

4. Завершите merge
   ```bash
   git commit -m "Merge: разрешены конфликты с main"
   ```

5. Отправьте обновления
   ```bash
   git push origin your-branch
   ```

## 10. Часто задаваемые вопросы

**В: Как начать с первого вклада?**

О: Начните с Issue помеченного `good first issue`. Это проблемы, которые хороши для новичков.

**В: Могу ли я работать над несколькими Issue одновременно?**

О: Да, но рекомендуется работать над ними в отдельных ветках и отправлять отдельные Pull Requests.

**В: Что делать, если мой PR не принят?**

О: Это нормально. Получите обратную связь, внесите улучшения и попробуйте снова.

**В: Нужно ли перед PR обсуждать идею?**

О: Для больших изменений рекомендуется сначала открыть Issue для обсуждения.

**В: Как долго ждать ответа на PR?**

О: Обычно рецензенты отвечают в течение 1-2 недель.
